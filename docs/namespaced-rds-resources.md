# Namespaced RDS Resources (Crossplane v2 Support)

## Overview

This provider ships namespaced (scope=Namespaced) variants of all RDS managed resources alongside the original cluster-scoped ones. This enables usage with Crossplane v2 features: namespaced XRDs, ManagedResourceActivationPolicies (MRAP), and namespace-scoped ProviderConfigs.

**Resources covered:** DBInstance, DBCluster, DBParameterGroup, DBClusterParameterGroup, DBInstanceRoleAssociation, GlobalCluster, OptionGroup, ResourceRecordSet, DBSubnetGroup, SecurityGroup.

## Problem Statement

Crossplane v2 expects managed resources to be namespaced with a specific spec shape (`ManagedResourceSpec` from `crossplane-runtime/v2`). Our provider uses `crossplane-runtime v1.20.x` which only produces cluster-scoped CRDs. A full migration to crossplane-runtime/v2 is non-trivial, so we implemented a temporary solution that produces v2-compatible namespaced CRDs while keeping the v1 runtime.

## Approaches Tried

### Attempt 1: Import crossplane-runtime/v2 and embed ManagedResourceSpec (FAILED)

**What was done:**
- Added `github.com/crossplane/crossplane-runtime/v2 v2.2.2` to `go.mod`
- Created `NamespacedSpec` embedding `xpv2.ManagedResourceSpec` from `crossplane-runtime/v2/apis/common/v2`
- Created `NamespacedStatus` embedding `xpcommon.ResourceStatus` from `crossplane-runtime/v2/apis/common`

**Why it failed:**
- `crossplane-runtime/v2 v2.2.2` requires `sigs.k8s.io/controller-runtime v0.23.1`
- Go module resolution picks the highest required version, upgrading controller-runtime from v0.19.0 to v0.23.1
- `crossplane-runtime v1.20.x` does not compile against controller-runtime v0.23.1:
  - `controller-runtime v0.23.1` added `Apply(ctx, runtime.ApplyConfiguration, ...ApplyOption)` to `client.SubResourceWriter` — the v1 runtime's `unstructured.wrapperStatusClient` doesn't implement it
  - `resource.ClientApplicator` (which embeds both `client.Client` and `Applicator`) gets an ambiguous `Apply` selector because both embedded types now define an `Apply` method with different signatures
- These are compile-time failures with no workaround short of patching vendored code

**What was reverted:**
- Removed `crossplane-runtime/v2` from `go.mod`
- Downgraded `controller-runtime` back to v0.19.0
- `go mod tidy` + `go mod vendor`

### Attempt 2: Patch vendor to fix controller-runtime incompatibility (REJECTED)

**What was considered:**
- Adding the missing `Apply` method to `vendor/github.com/crossplane/crossplane-runtime/pkg/resource/unstructured/client.go`
- Disambiguating the `Apply` call in `vendor/github.com/crossplane/crossplane-runtime/pkg/connection/store/kubernetes/store.go`

**Why it was rejected:**
- Vendor patches are fragile — they get lost on `go mod vendor` and are invisible to reviewers
- The cascade of fixes kept growing (each fix exposed the next incompatibility)
- Not maintainable long-term

### Attempt 3: Inline the v2 spec fields without importing v2 (ADOPTED)

**Insight:** Crossplane reconciles CRDs based on their JSON schema, not Go import paths. If the CRD YAML has the right field structure, Crossplane v2 treats it as a proper namespaced managed resource regardless of how the Go code produces it.

**Solution:** Define the 3 fields from `ManagedResourceSpec` inline with identical JSON tags. No v2 import needed.

### controller-gen Workaround

**Problem:** `controller-gen v0.16.0` panics at `pkg/deepcopy/traverse.go:623` when processing types that embed structs from external packages it can't fully resolve. The `+kubebuilder:object:root=true` marker triggers the deepcopy traversal.

**Solution:** Removed both `+kubebuilder:object:generate=true` (package-level) and `+kubebuilder:object:root=true` (type-level) from v1alpha1ns. DeepCopy is generated by `hack/gen-namespaced.sh` instead.

**Impact:** CRD generation still works (uses `+kubebuilder:resource`, `+kubebuilder:subresource:status`, and `+kubebuilder:printcolumn` markers). Only deepcopy generation is skipped by controller-gen.

**Future fix:** Upgrading `sigs.k8s.io/controller-tools` beyond v0.16.0 should fix the panic, allowing the markers to be restored and manual deepcopy to be dropped.

### sed Patching Logic

Controllers are generated by copying files and applying:

```bash
# Change package name
s/^package dbinstance$/package dbinstance_ns/

# Add nsapitypes import alongside existing svcapitypes
s|svcapitypes "...v1alpha1"|svcapitypes "...v1alpha1"\n\tnsapitypes "...v1alpha1ns"|

# Replace root CRD type references (uses [^A-Za-z0-9_] boundary, not \b which is broken on macOS sed)
s|*svcapitypes.DBInstance([^A-Za-z0-9_])|*nsapitypes.DBInstance\1|g
s|&svcapitypes.DBInstance{}|&nsapitypes.DBInstance{}|g
s|svcapitypes.DBInstanceGroupKind|nsapitypes.DBInstanceGroupKind|g
s|svcapitypes.DBInstanceGroupVersionKind|nsapitypes.DBInstanceGroupVersionKind|g
```

Helper types (`svcapitypes.Tag`, `svcapitypes.Endpoint`, etc.) are NOT replaced — they're the same types shared between packages.

**Note on macOS sed:** `\b` (word boundary) doesn't work in macOS `sed`. The initial attempt used `\b` and silently failed to replace function return types, causing type mismatches. Fixed by using `[^A-Za-z0-9_]` as a boundary pattern instead.

## Regenerating After ACK Changes

After running `make services SERVICES=rds`:

```bash
./hack/gen-namespaced.sh
```

**What the script regenerates:** All files in `apis/rds/v1alpha1ns/` and `pkg/controller/rds/*_ns/` (except `helpers.go`).

**What requires manual attention after regeneration:**
- `pkg/controller/rds/dbinstance_ns/helpers.go` — If ReadReplica logic changes.
- Any new custom hooks in `setup.go` that pass `*svcapitypes.DBInstance` to functions expecting the cluster-scoped type.
- Unused import cleanup (the script's verification step will report these).

## Limitations

- **Drift risk** — `_ns` controller copies are stale if originals change. Re-run the script.
- **No reference resolvers** — Namespaced types lack `ResolveReferences()`. Use composition or manual resolution.
- **No dedicated tests** — Relies on same logic as cluster-scoped controllers.
- **DeletionPolicy supported** — Unlike the upstream v2 `ManagedResourceSpec` (which omits `DeletionPolicy`), we include it. Users can set `deletionPolicy: Orphan` to keep the AWS resource when deleting the CR. Default is `Delete`.
- **PublishConnectionDetailsTo not supported** — v2 uses only `writeConnectionSecretToRef` (local, same namespace).
- **WriteConnectionSecretToReference** — The bridge uses the CR's namespace when returning a `SecretReference` to the reconciler.
- **Only cluster-scoped ProviderConfig** — The `providerConfigRef` field accepts a name only (no `kind`). The controller always resolves a cluster-scoped `ProviderConfig`. Namespace-local ProviderConfigs are not supported in this temporary implementation.

## RBAC

The controller needs RBAC permissions for the new API group `rds.aws.m.crossplane.io`. Verify that the deployed ClusterRole includes rules like:

```yaml
- apiGroups: ["rds.aws.m.crossplane.io"]
  resources: ["*"]
  verbs: ["*"]
- apiGroups: ["rds.aws.m.crossplane.io"]
  resources: ["*/status"]
  verbs: ["get", "patch", "update"]
```

If using Crossplane's package manager, these rules should be generated from the CRD YAMLs in `package/crds/`. If deploying manually, add them to the provider's ClusterRole.

## CRD Installation

CRD YAMLs are generated in `package/crds/rds.aws.m.crossplane.io_*.yaml` by `controller-gen crd` during `make generate`. This works because the `+kubebuilder:resource:scope=Namespaced` marker is present on the types (the missing `+kubebuilder:object:root=true` only affects deepcopy generation, not CRD generation).

To verify CRDs are generated:
```bash
ls package/crds/rds.aws.m.crossplane.io_*
```

For local testing, apply them manually:
```bash
kubectl apply -f package/crds/rds.aws.m.crossplane.io_dbinstances.yaml
```

When deployed as a Crossplane package, ManagedResourceActivationPolicy (MRAP) controls which CRDs are active.
